# v0.5 Editor Extension Delivery Plan

This plan defines the v0.5 work that turns the local LSP/editor verification
setup into a publishable VS Code-compatible extension for VS Code, Cursor, and
Open VSX-compatible editors.

The current implementation already provides the core runtime pieces:

- `docbridge lsp` exposes the link graph through LSP over stdio.
- `editors/vscode` contains a VS Code-compatible client extension.
- `just verify-lsp` exercises Hover, Definition, References, and Diagnostics
  without an editor.
- `just vscode-lsp` and `just cursor-lsp` install the local development VSIX
  into VS Code or Cursor for source-checkout verification.

v0.5 productizes that extension path. MCP delivery is intentionally out of scope
for this project until a concrete consumer need appears.

Normative behavior is reflected in:

- [CLI](../../specs/cli.md)
- [LSP](../../specs/lsp.md)
- [VS Code Client README](../../../editors/vscode/README.md)

## Goals

- Keep `docbridge lsp` as the editor-facing server protocol.
- Publish the extension identity as `salan70.docbridge`.
- Build a universal VSIX that bundles the DocBridge CLI/server runtime.
- Keep Bun as an external runtime requirement for the first extension release.
- Include Swift and Dart document activation alongside TypeScript, TSX, and
  Markdown.
- Require `darwin-arm64` and `linux-x64` scanner binaries in the VSIX package.
- Verify the exact VSIX artifact before publishing.
- Publish manually to VS Code Marketplace and Open VSX for the first release.
- Leave release-workflow automation and Zed-specific work as follow-up issues.

## Non-Goals

- MCP server or MCP-based agent delivery.
- Zed-specific packaging or extension work.
- CI/CD automation for registry publishing or GitHub Release VSIX attachment.
- Bundling Bun into the VSIX.
- Windows scanner support in the first VSIX.
- Replacing the existing local `just vscode-lsp` / `just cursor-lsp`
  development flows.

## Decisions

### Local Verification vs Release Artifact

Local verification commands remain development tools:

```sh
just verify-lsp
just vscode-lsp
just cursor-lsp
```

They may compile from source, use local workspace settings, install development
dependencies, and open the repository checkout.

The release path produces a separately verifiable VSIX:

```sh
just package-vsix
just verify-vsix
```

The VSIX is written under `editors/vscode/.tmp/out/` and can be installed into
VS Code or Cursor, attached manually to a GitHub Release, or published manually
to extension registries.

### Extension Identity and Metadata

The public extension ID is `salan70.docbridge`:

- `publisher`: `salan70`
- `name`: `docbridge`
- `displayName`: `DocBridge`

The extension version must match the root DocBridge package version. The release
packaging command fails if the versions drift. The extension references
`editors/vscode/assets/icon.png`; packaging fails until that icon exists.

### Bundled Server

The VSIX bundles a runtime subset of the root package:

- `package.json`
- `dist/`
- `schemas/`
- `templates/skills/`
- `README.md`
- `CHANGELOG.md`
- `LICENSE`

The extension starts the bundled server as:

```text
bun server/dist/index.js lsp
```

`docbridge.cliPath` remains available for source-checkout development overrides,
but it must be an absolute path.

### Supported Documents

The extension attaches to these VS Code language IDs:

- `typescript`
- `typescriptreact`
- `swift`
- `dart`
- `markdown`

The extension does not duplicate DocBridge include-pattern filtering. It only
selects candidate document languages; the server decides which files are managed
from `docbridge.config.json`.

### Manual Publishing

The first registry publish is manual:

```sh
VSCE_PAT=<token> just publish-vscode-extension
OVSX_PAT=<token> just publish-open-vsx-extension
```

Both commands publish the already-built VSIX. They also accept an explicit VSIX
path so the same verified artifact can be reused:

```sh
VSCE_PAT=<token> just publish-vscode-extension path/to/docbridge.vsix
OVSX_PAT=<token> just publish-open-vsx-extension path/to/docbridge.vsix
```

Automation is deferred until the manual flow proves stable.

## Follow-up Work

- Add release workflow automation for VSIX attachment and registry publishing.
- Decide whether Zed should use direct LSP configuration, an extension package,
  or another delivery path.
- Revisit MCP only if a concrete agent integration needs a long-lived tool
  server rather than the existing CLI, hooks, and skills.
