# v0.5 Editor and Agent Delivery Plan

This plan defines the v0.5 delivery work that turns the current local
LSP/editor verification setup into productized delivery channels for editors
and AI agents.

The current implementation already provides the core runtime pieces:

- `docbridge lsp` exposes the link graph through LSP over stdio.
- `editors/vscode` contains a VS Code-compatible client extension.
- `just verify-lsp` exercises Hover, Definition, References, and Diagnostics
  without an editor.
- `just vscode-lsp` packages and installs the current VS Code-compatible
  extension into VS Code for local verification.
- `just cursor-lsp` installs the same extension into Cursor for local
  verification.

Those commands prove the server and editor client locally. They are not yet a
supported delivery story for end users. v0.5 draws that boundary explicitly and
adds release artifacts, publication decisions, and MCP-based agent delivery.

Normative behavior will be reflected in these specs and docs as the slices
land:

- [CLI](../specs/cli.md)
- [LSP](../specs/lsp.md)
- [Claude Code Integration](../integrations/claude-code.md)
- [Codex Integration](../integrations/codex.md)
- [VS Code Client README](../../editors/vscode/README.md)

## This Planning PR

The PR for this issue should only add this plan. It should not productize the
extension, publish a VSIX, add MCP runtime code, or rewrite public README files.
Those changes are intentionally split into the slices below.

## Goals

- Keep `docbridge lsp` as the editor-facing server protocol.
- Turn the existing VS Code-compatible extension from a local verification
  package into a releasable VSIX artifact.
- Preserve Cursor support as a VS Code-compatible extension target instead of
  inventing a Cursor-specific extension.
- Decide whether the first public editor channel is direct VSIX download, VS
  Code Marketplace, Open VSX, or a staged combination of those channels.
- Define and ship an MCP server that exposes DocBridge data to agent clients.
- Reuse the existing graph, context, and related-gate implementation paths
  instead of adding a parallel scanner or resolver.
- Keep editor and agent dependencies isolated from the CLI core where possible.
- Keep each delivery slice independently reviewable and testable.

## Non-Goals

- Replacing `docbridge lsp` with an editor-specific protocol.
- Implementing Zed-specific support in the first delivery slice.
- Rewriting the main README, Japanese README, or integration docs before the
  relevant delivery channel actually exists.
- Bundling a custom Bun runtime into the first VSIX unless a later slice proves
  that requiring an external Bun runtime is not acceptable.
- Adding a web service, daemon, or network server. Both LSP and MCP should start
  as stdio processes.
- Solving large-project incremental re-resolution or multi-root workspaces.

## Decisions

### Local Verification vs Distributable Support

Local verification commands remain development tools:

```sh
just verify-lsp
just vscode-lsp
just cursor-lsp
```

They may compile from source, use local workspace settings, install development
dependencies, and open the repository checkout.

Distributable support requires a separately verifiable artifact:

- a VSIX that can be installed outside this checkout,
- extension metadata suitable for the chosen publication channel,
- a bundled or clearly resolved DocBridge server entry point,
- smoke tests that install the built artifact into VS Code and Cursor, and
- documentation that describes the supported install path rather than the local
  development path.

The current `scripts/package-editor-vsix.sh` script is a useful starting point,
but it is still development-oriented: it copies `src/`, editor dependencies,
and local package files. v0.5 should introduce a release artifact path that is
based on the same built runtime artifacts used by package distribution.

### VS Code-Compatible Extension

The `editors/vscode` package remains the single editor client package for VS
Code-compatible editors. It should continue to hold the `vscode-languageclient`
dependency so `src/core/` and the LSP server stay dependency-minimal.

The productized extension needs explicit metadata before publication:

- stable extension ID and publisher ID,
- display name and description,
- repository, issue, license, and homepage links,
- categories, keywords, and supported languages,
- icon and marketplace assets if required by the target channel,
- version alignment with the DocBridge CLI release process,
- bundled server layout, and
- user-facing settings for Bun and optional CLI overrides.

The first release can still require Bun, but the startup failure must be clear
when Bun is missing.

### Cursor Target

Cursor support should be verified as a VS Code-compatible target. The
implementation should avoid Cursor-specific code unless the compatibility test
finds a real divergence.

`just cursor-lsp` remains the local development path. v0.5 should add a release
artifact verification path that installs the built VSIX into Cursor and proves
that the extension starts `docbridge lsp` successfully.

### Marketplace and Open VSX

Marketplace and Open VSX publishing should be decided before automation is
added. The decision slice should answer:

- whether the first public channel is direct GitHub Release VSIX attachment,
  VS Code Marketplace, Open VSX, or multiple channels,
- which publisher identities and tokens are required,
- whether Marketplace and Open VSX packages can share the same VSIX,
- what metadata each registry requires,
- whether publishing runs from the existing release workflow or a separate
  manually triggered workflow, and
- how failed editor publishing affects npm release success.

Until this decision lands, release artifact generation can be implemented
without uploading to a registry.

### MCP Server

The MCP server should expose existing DocBridge data as tools for agent
clients. It should not introduce a separate project model.

The first tool contract should cover:

- link graph access, equivalent to `docbridge graph --json`,
- counterpart context access, equivalent to `docbridge context --json`, and
- gate-oriented related file triage if the agent recipes need the same behavior
  as `docbridge related --gate --json`.

The server should run over stdio and be launched through the DocBridge CLI,
for example as a future `docbridge mcp` command if the command contract slice
confirms that shape. The command should be packaged with the npm CLI and use
the same root resolution, config loading, scanners, and output schemas as the
existing CLI commands.

### Agent Delivery

MCP should complement the current hook and skill recipes, not replace them in
the first slice. Existing integrations for Claude Code and Codex already
document hooks, skills, `docbridge context`, and `docbridge related --gate`.
v0.5 should add MCP recipes once the server contract exists.

Agent-specific documentation should stay concrete:

- how to launch the MCP server from an installed `docbridge` package,
- which tools are available,
- what project root the server uses,
- how results map back to existing CLI JSON schemas, and
- which existing hook/skill flows remain useful.

### Zed and Public Docs Follow-up

Zed-specific work should be a later issue. Zed may use LSP directly, an
extension package, an MCP path, or a combination, but that decision should be
made after the VS Code-compatible and MCP delivery paths are proven.

Public documentation updates should also follow the implementation slices. The
README and Japanese README should not claim installable editor or MCP support
until the artifact and server slices exist.

## Agent Workflow

AI agents should work one slice at a time. Each slice should leave the
repository in a working state.

For each slice:

1. Read the relevant spec, decision, or integration doc before editing.
2. Add or update focused tests or verification scripts before changing behavior.
3. Keep editor-package dependencies inside `editors/vscode` unless the slice
   explicitly justifies a CLI-level dependency.
4. Run `just check` and `just test`.
5. Run the slice-specific verification command listed below.
6. Keep public docs truthful to the artifact or server state that actually
   exists after the slice.

## Slice 1: Delivery Boundary and Extension Metadata

Purpose: make the current extension package ready to become a distributable
artifact without changing the runtime behavior yet.

Tasks:

- Update `editors/vscode/package.json` metadata for a public extension package.
- Decide whether `private: true` remains for local development or is removed
  only in the release packaging step.
- Add or document the stable extension ID and publisher ID.
- Add required metadata fields for VS Code-compatible publication.
- Keep the existing `docbridge.bunPath` and `docbridge.cliPath` settings.
- Document the difference between local verification and distributable support
  in the editor README.

Tests:

- Existing extension compile still succeeds.
- Metadata validation passes through the chosen package verification command or
  a focused script if registry tooling is not introduced yet.

Verification:

```sh
just check
just test
```

Done when:

- The extension package has enough stable metadata for release artifact work to
  proceed.
- The docs still describe the current artifact as local verification only.

## Slice 2: Release VSIX Build

Purpose: produce a release-oriented VSIX artifact that can run outside the
repository checkout.

Tasks:

- Add a release VSIX build path separate from or clearly parameterized against
  the current local installer path.
- Build the DocBridge CLI server artifact before packaging the VSIX.
- Bundle the server artifact and required runtime files in a deterministic
  layout.
- Avoid copying unnecessary source, tests, fixtures, or development outputs.
- Preserve the existing development VSIX flow used by `just vscode-lsp` and
  `just cursor-lsp`.
- Emit the VSIX path in a predictable location for CI and GitHub Release upload.

Tests:

- Package layout verification checks that required files are present and known
  development-only files are absent.
- Extension compile and VSIX generation run from a clean checkout.

Verification:

```sh
just check
just test
just build
```

Done when:

- A deterministic release VSIX can be built locally and in CI.
- The artifact starts from bundled DocBridge runtime files rather than a source
  checkout assumption.

## Slice 3: VS Code Release Artifact Verification

Purpose: prove the release VSIX works in VS Code.

Tasks:

- Add a verification command that installs the release VSIX into VS Code.
- Open a DocBridge fixture or example workspace.
- Confirm that `docbridge lsp` starts from the packaged server path.
- Exercise Diagnostics, Hover, Definition, and References through the installed
  extension, or document any manual step that cannot be automated locally.
- Keep local developer installation commands available.

Tests:

- Automated checks should cover the installable artifact as far as the local
  editor CLI allows.
- Existing `just verify-lsp` remains editor-independent and must keep passing.

Verification:

```sh
just verify-lsp
```

Done when:

- The release VSIX has a repeatable VS Code smoke test.
- Startup failures report actionable messages in the DocBridge output channel.

## Slice 4: Cursor Compatibility Verification

Purpose: prove the same release VSIX works in Cursor.

Tasks:

- Add or extend a verification command that installs the release VSIX into
  Cursor.
- Confirm that Cursor can activate the same VS Code-compatible extension.
- Capture any Cursor-specific setting, command, or output-channel difference in
  the editor README.
- Avoid Cursor-specific code unless the smoke test demonstrates a real need.

Tests:

- Cursor verification reuses the same release VSIX generated for VS Code.
- `just cursor-lsp` remains available for local development.

Verification:

```sh
just verify-lsp
```

Done when:

- Cursor is a verified VS Code-compatible target for the built VSIX.
- Any Cursor-specific operational notes are documented.

## Slice 5: Marketplace and Open VSX Publishing Decision

Purpose: decide the public extension publication path before adding registry

Tasks:

- Document required publisher accounts, tokens, and repository secrets.
- Decide whether registry publishing is part of the normal release workflow.
- Decide whether GitHub Releases should always attach the VSIX.
- Decide how editor artifact failures interact with npm package release

- If publishing automation is in scope for the same slice, add it only after the

Tests:

Verification:


## Slice 6: MCP Tool Contract

Purpose: define the MCP surface before implementing runtime code.

Tasks:

- Decide the CLI entry point shape, expected to be `docbridge mcp` unless the
  slice finds a better command contract.
- Define tool names, inputs, outputs, and error behavior.
- Reuse existing JSON schema concepts from `graph`, `context`, and `related`.
- Decide whether the MCP implementation uses an official SDK dependency or a
  small local stdio implementation, and document the dependency tradeoff.
- Add contract tests or schema fixtures that can drive the implementation
  slice.
- Update the CLI spec with the new command only if the command shape is final.

Tests:

- Contract fixtures validate expected request and response shapes.
- Existing CLI commands remain unchanged.

Verification:

```sh
just check
just test
```

Done when:

- The MCP command and tool contract are precise enough for an implementation PR.

## Slice 7: MCP Server Command

Purpose: implement the MCP server delivery path.

Tasks:

- Implement the selected MCP command.
- Wire MCP tools to existing core functions for graph, context, and related gate
  behavior.
- Share root resolution and config loading behavior with existing CLI commands.
- Keep scanner and resolver behavior identical to the CLI.
- Add focused tests for tool success and failure responses.
- Ensure package builds include the MCP command path.

Tests:

- Unit or integration tests drive MCP requests over stdio.
- CLI help and unknown-command behavior stay correct.
- Existing graph, context, related, check, and LSP tests keep passing.

Verification:

```sh
just check
just test
just build
```

Done when:

- An installed DocBridge package can launch the MCP server and serve the planned
  tools.
