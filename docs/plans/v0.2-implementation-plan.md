# v0.2 Implementation Plan

This plan breaks SpecLink v0.2 into implementation slices that are suitable for
AI coding agents. Each slice should be small enough to review independently and
should leave the repository in a working state.

Normative behavior lives in:

- [LSP](../specs/lsp.md)
- [Diagnostics](../specs/diagnostics.md)
- [v0.2 Decisions](../decisions/v0.2.md)

v0.2 is additive. `speclink check` and the v0.1 specs are unchanged.

## Status

- [x] Slice 1: LSP Transport Foundation
- [x] Slice 2: Project Model and Link Graph
- [x] Slice 3: Diagnostics over LSP
- [x] Slice 4: Position Index and Range Enrichment
- [x] Slice 5: Hover

- [x] Slice 9: Alignment and Release Readiness

## Goals

- Implement `speclink lsp`, a Language Server over stdio, without an LSP
  framework library.
- Provide `publishDiagnostics`, Hover, Definition, and References over the
  SpecLink link graph.
- Reuse the v0.1 scanners and resolver; expose the resolved link graph for
  navigation.
- Verify the server in VS Code through a minimal, isolated client extension.
- Keep the server and `src/core/` dependency-minimal.

## Non-Goals

- Standalone single-binary distribution and npm publishing.
- Editors other than VS Code as a verified target; Marketplace publishing.
- Incremental re-resolution and large-project performance work.
- Multi-root / monorepo workspaces.
- Code intelligence other language servers already provide (types, completion,
  formatting).

## Agent Workflow

AI agents should work one slice at a time.

For each slice:

1. Read the relevant spec before editing code.
2. Add or update focused Bun tests first. For logic changes, use the `tdd` skill.
3. Implement the minimum production code required by the tests.
4. Run `just test`.
5. Run the slice-specific verification commands listed below.
6. Keep unrelated formatting, refactors, and generated output out of the diff.

Keep `src/core/` pure and the LSP transport in a dedicated module. The VS Code
extension lives in its own package and must not leak its dependency into the
server.

## Proposed Module Layout

This layout is not mandatory, but agents should avoid inventing overlapping
modules without a clear reason.

```text
src/
  cli/
    index.ts
  core/
    ... (v0.1 modules, plus the exposed link graph)
  lsp/
    server.ts        # lifecycle and request dispatch
    transport.ts     # JSON-RPC over stdio framing
    documents.ts     # open-document buffers and overlay
    project.ts       # whole-project scan, overlay, re-resolve
    index-lookup.ts  # position -> endpoint hit testing
    hover.ts
    definition.ts
    references.ts
    diagnostics.ts   # SpecLinkDiagnostic -> LSP Diagnostic
editors/
  vscode/            # minimal client extension (isolated dependency)
```

Expected test files:

```text
test/
  lsp/
    transport.test.ts
    lifecycle.test.ts
    project.test.ts
    diagnostics.test.ts
    index-lookup.test.ts
    hover.test.ts
    definition.test.ts
    references.test.ts
    conformance.test.ts   # child-process initialize -> didOpen -> request
```

## Slice 1: LSP Transport Foundation

Purpose: a server that speaks LSP framing and completes the lifecycle.

Tasks:

- Implement JSON-RPC over stdio with `Content-Length` framing, using UTF-8 byte
  length.
- Handle bodies split across reads and multiple messages in one read.
- Implement `initialize` (declare capabilities, resolve root from `rootUri` /
  `workspaceFolders`), `initialized`, `shutdown`, `exit`.
- Declare Full `textDocumentSync` and the hover, definition, and references
  capabilities.
- Wire `speclink lsp` to start the server.

Tests:

- Frame a message and parse it back (round-trip), including a multi-byte UTF-8
  body.
- Parse a buffer containing two concatenated messages.
- Parse a body delivered in two reads.
- `initialize` returns the declared capabilities.

Verification:

```sh
just test
```

Done when:

- `speclink lsp` completes an `initialize` / `initialized` / `shutdown` / `exit`
  cycle and returns correct capabilities.

## Slice 2: Project Model and Link Graph

Purpose: build and maintain the whole-project link graph with buffer overlay.

Tasks:

- On `initialized`, load config, collect include-matched files, scan from disk,
  and resolve the graph using the v0.1 pipeline.
- Maintain open-document buffers; overlay buffer text over disk content per URI.
- Re-resolve the project on change, with a short debounce.
- Expose the resolved link graph (endpoint-to-endpoint pairs with locations) as a
  queryable structure for later slices.

Tests:

- Graph built from disk matches the resolver output for a fixture root.
- A buffer overlay overrides the on-disk content for an open URI.
- `didClose` reverts the URI to its on-disk version.

Verification:

```sh
just test
```

Done when:

- The server holds a correct whole-project graph and updates it on change.

## Slice 3: Diagnostics over LSP

Purpose: surface SpecLink diagnostics in the editor.

Tasks:

- Map `SpecLinkDiagnostic` to LSP `Diagnostic` (severity, range, code, message)
  per [Diagnostics](../specs/diagnostics.md).
- Publish diagnostics per open URI via `textDocument/publishDiagnostics`.
- Refresh published diagnostics after a debounced re-resolution.

Tests:

- `didOpen` on a file with errors publishes the expected diagnostic shape.
- A change that fixes an error clears it on the next publish.
- Severity and code mapping is correct.

Verification:

```sh
just test
```

Done when:

- Opening a file with link problems shows correct diagnostics over LSP.

## Slice 4: Position Index and Range Enrichment

Purpose: map editor positions to endpoints.

Tasks:

- Enrich the scanners to record `nameRange` (code), `headingTextRange`
  (Markdown), and `targetRange` (annotation target string).
- Build a position-to-endpoint lookup over the graph.
- Convert between LSP 0-based UTF-16 positions and SpecLink 1-based positions.

Tests:

- A position within a name / heading resolves to the endpoint.
- A position on whitespace, parameters, or `#` markers does not resolve.
- 0-based / 1-based conversion at line and column boundaries.

Verification:

```sh
just test
```

Done when:

- Hit testing matches the spec's trigger zones.

## Slice 5: Hover

Purpose: Dart-like inline documentation on hover.

Tasks:

- Code to doc: extract the linked Markdown section (heading to next same-or-higher
  heading, fence-aware) and return it as Markdown hover content.
- Doc to code: return the linked endpoint plus the declaration signature line.
- Concatenate one-to-many sections; apply the length cap.

Tests:

- Section extraction includes deeper subsections and stops at the next
  same-or-higher heading.
- A `#` inside a fenced block does not end the section.
- Doc-to-code hover shows the signature line.
- One-to-many concatenation.

Verification:

```sh
just test
```

Done when:

- Hover over a linked symbol shows the linked spec section.

## Slice 6: Definition

Purpose: jump to the linked counterpart.

Tasks:

- Return counterpart `Location`(s) for the element under the cursor.
- Return multiple `Location`s for one-to-many links.

Tests:

- Code-to-doc jump returns the heading location.
- Doc-to-code jump returns the declaration location.
- One-to-many returns multiple locations.

Verification:

```sh
just test
```

Done when:

- Go to Definition crosses between code and docs per the spec.

## Slice 7: References

Purpose: list all linked counterparts.

Tasks:

- Return every counterpart linked to the element (symmetric counterpart model).

Tests:

- A heading referenced by many symbols returns all of them.
- A symbol returns the doc section(s) it links to.

Verification:

```sh
just test
```

Done when:

- Find All References lists all counterparts per the spec.

## Slice 8: VS Code Client Extension

Purpose: verify the server in a real editor.

Tasks:

- Add a minimal client extension under `editors/vscode/` using
  `vscode-languageclient`.
- Launch `speclink lsp` and bind it to TypeScript and Markdown documents.
- Keep the `vscode-languageclient` dependency isolated to this package; the
  server and `src/core/` must not depend on it.

Tests:

- Manual verification under the Extension Development Host (F5): Diagnostics,
  Hover, Definition, and References against a TypeScript and Markdown pair.

Verification:

```sh
just test
# Manual: open editors/vscode in VS Code, press F5, exercise the four features.
```

Done when:

- The four features work in the Extension Development Host.

## Slice 9: Alignment and Release Readiness

Purpose: confirm v0.2 builds and the docs match behavior.

Tasks:

- Add a protocol conformance test that drives the server as a child process.
- Ensure `bun build` succeeds and `speclink lsp` runs.
- Review README, `docs/ja/README.md`, schema, and specs against behavior.
- Bump the version to `0.2.0` at release time.

Verification:

```sh
just test
just check
just build
```

Done when:

- All repo-native commands pass.
- README, specs, and the conformance test agree with implemented behavior.

## Review Checklist

Before merging a slice, check:

- The implementation follows the relevant spec exactly, or the spec was updated
  in the same change with a clear rationale.
- Tests cover both the success path and at least one failure path.
- The slice does not add runtime dependencies to the server or `src/core/`. The
  only permitted new dependency is `vscode-languageclient`, confined to
  `editors/vscode/`.
- `just test` passes.
- Any changed behavior is reflected in README or specs when user-facing.

## Risks

- JSON-RPC framing edge cases (UTF-8 byte length, split reads, batched messages)
  are easy to get subtly wrong. Cover them with focused transport tests.
- Single-point locations make hit testing imprecise. Range enrichment (Slice 4)
  is a prerequisite for usable Hover, Definition, and References.
- Whole-project re-resolution on every change is acceptable at this scale but
  needs a debounce. Incremental re-resolution is deferred.
- VS Code cannot launch an arbitrary LSP server without a client extension. The
  extension is required for in-editor verification and pulls a small amount of
  v0.4 scope forward; keep it minimal and isolated.
