# v0.1 Implementation Plan

This plan breaks SpecLink v0.1 into implementation slices that are suitable for
AI coding agents. Each slice should be small enough to review independently and
should leave the repository in a working state.

Normative behavior lives in:

- [Configuration](../specs/configuration.md)
- [Scanning](../specs/scanning.md)
- [Annotations](../specs/annotations.md)
- [Link Resolution](../specs/link-resolution.md)
- [Diagnostics](../specs/diagnostics.md)
- [CLI](../specs/cli.md)
- [v0.1 Decisions](../decisions/v0.1.md)

## Goals

- Implement `speclink check` for small TypeScript and Markdown projects.
- Support this repository and `examples/basic` as real scan roots.
- Keep the implementation dependency-light: Bun, TypeScript, and the TypeScript
  Compiler API.
- Make diagnostics deterministic and machine-readable.
- Use test-first implementation for parsing and resolution behavior.

## Non-Goals

- LSP features such as hover, definition, and references.
- `.tsx`, `.d.ts`, Setext headings, file-only links, fragment-only links, and
  external URLs.
- `.gitignore` support or custom exclude configuration.
- Schema fetching or JSON Schema validation at runtime.
- MCP server or editor integration.

## Agent Workflow

AI agents should work one slice at a time.

For each slice:

1. Read the relevant spec before editing code.
2. Add or update focused Bun tests first.
3. Implement the minimum production code required by the tests.
4. Run `just test`.
5. Run the slice-specific verification commands listed below.
6. Keep unrelated formatting, refactors, and generated output out of the diff.

Prefer pure functions in `src/core/` and keep `src/cli/` responsible for argument
parsing, output formatting, and process exit behavior.

## Proposed Module Layout

This layout is not mandatory, but agents should avoid inventing overlapping
modules without a clear reason.

```text
src/
  cli/
    index.ts
  core/
    config.ts
```

## Slice 2: Link Target Parsing

Purpose: centralize `file#fragment` validation so Markdown, TypeScript, and
resolution use the same rules.

Tasks:

- Parse project-root-relative targets.
- Reject empty file paths, empty fragments, multiple `#` characters, whitespace
  inside targets, absolute paths, `./`, `../`, and backslash separators.
- Preserve raw fragments without URL decoding.
- Emit `invalid_link_target` with source location when available.

Tests:

- Accept `docs/specs/cli.md#check-command`.
- Reject same-file targets, fragment-only targets, file-only targets, absolute
  paths, traversal, backslashes, whitespace, and multiple fragments.

Verification:

```sh
just test
```

Done when:

- All annotation consumers can call one target parser.

## Slice 3: Markdown Scanner

Purpose: extract Markdown headings, anchors, and `@code` annotations.

Tasks:

- Recognize ATX headings indented by 0 to 3 spaces.
- Generate anchors using the v0.1 rules.
- Ignore Setext headings.
- Recognize fenced code blocks opened by backticks or tildes and ignore headings
  and comments inside them.
- Recognize standalone HTML comments whose trimmed body starts with `@code`.
- Attach pending `@code` comments to the next non-empty heading.
- Allow empty lines between pending `@code` comments and the next heading.
- Emit `dangling_code_annotation` when pending comments are invalidated or attach
  to an empty heading.
- Emit `duplicate_doc_anchor` for duplicate non-empty anchors in one file.
- Emit `duplicate_link` for duplicate `@code` links from the same heading to the
  same code endpoint.

Tests:

- Anchor generation for punctuation, whitespace, case, Unicode letters, and
  optional closing `#`.
- Ignoring indented comments with 4 or more spaces.
- Ignoring fenced content.
- Multiple `@code` comments for one heading.
- Dangling annotations before normal text or non-`@code` comments.
- Duplicate anchors in the same file only.

Verification:

```sh
just test
```

Done when:

- `examples/basic/docs/auth.md` produces one doc anchor and one `doc -> code`
  link.

## Slice 4: TypeScript Scanner

Purpose: extract exported TypeScript declarations and `@doc` annotations using
the TypeScript Compiler API.

Tasks:

- Parse `.ts` files and report `typescript_parse_error` on syntactic errors.
- Ignore `.d.ts` files.
- Extract top-level exported supported declarations:
  `function`, `class`, `abstract class`, `interface`, `type`, single-declarator
  `const`, `enum`, `const enum`, named default `function`, named default `class`,
  and supported `declare` forms.
- Extract `@doc` tags associated with supported declarations.
- Emit `unsupported_declaration` for unsupported declarations that have `@doc`.
- Ignore unsupported declarations without `@doc`.
- Emit `duplicate_code_symbol` for duplicate annotated code endpoints in the same
  file.
- Emit `duplicate_link` for duplicate `@doc` links from the same code endpoint to
  the same doc endpoint.

Tests:

- One test per supported declaration form.
- Unsupported annotated examples listed in `docs/specs/annotations.md`.
- Multiple `@doc` tags on one declaration.
- Named default exports.
- Syntactic parse errors suppress extraction from the affected file.

Verification:

```sh
just test
```

Done when:

- `examples/basic/src/auth/login.ts` produces one code symbol and one
  `code -> doc` link.

## Slice 5: Configuration and Glob Scanning

Purpose: load `speclink.config.json`, validate include patterns, and collect
managed files.

Tasks:

- Resolve project root from CLI options.
- Load optional `speclink.config.json`.
- Use default config when the file does not exist.
- Validate required `include.code` and `include.docs` arrays.
- Allow optional string `$schema`.
- Reject unknown keys and invalid values with config diagnostics.
- Implement v0.1 glob syntax: `*` and `**` only.
- Ignore `node_modules`, `.git`, dot-prefixed path segments, symlink files, and
  symlink directories.
- Exclude `.d.ts` files from code scans.
- Emit `file_read_error` for unreadable managed files.

Tests:

- Default config.
- Valid root config and valid example config.
- Unknown top-level and `include` keys.
- Invalid glob syntax and invalid path forms.
- Matching behavior for `*`, `**`, `.ts`, `.md`, and `.d.ts`.
- Built-in ignored paths.

Verification:

```sh
just test
```

Done when:

- The root repository config scans `src/**/*.ts` and `docs/specs/**/*.md`.
- `examples/basic` scans its own code and docs.

## Slice 6: Link Resolution

Purpose: combine scanner outputs and emit relationship diagnostics.

Tasks:

- Resolve `@doc` targets against managed Markdown files and anchors.
- Resolve `@code` targets against managed TypeScript files and matching `@doc`

- Emit `doc_file_not_found`.
- Emit `doc_anchor_not_found`.
- Emit `doc_backlink_not_found`.
- Emit `code_file_not_found`.

Tests:

- Valid bidirectional pair.
- Missing doc file.
- Missing doc anchor.
- Missing doc backlink.
- Missing code file.
- Missing code backlink.
- Derived diagnostics suppression.
- Audit-only undocumented symbols.

Verification:

```sh
just test
```

Done when:

## Recommended Agent Assignments

Use separate agent runs for slices that have clear boundaries:

- Agent A: Slices 1 and 2.
- Agent B: Slice 3.
- Agent C: Slice 4.
- Agent D: Slice 5.
- Agent E: Slice 6.
- Agent F: Slices 7 through 9.

Do not run agents concurrently until Slice 1 is complete, because shared types
and diagnostic helpers affect every later slice.

After Slice 1, Markdown and TypeScript scanner work can proceed in parallel if
both agents agree on the shared endpoint and diagnostic types. Configuration,
glob scanning, and CLI integration should wait until scanners expose stable
interfaces.

## Review Checklist

Before merging a slice, check:

- The implementation follows the relevant spec exactly, or the spec was updated
  in the same change with a clear rationale.
- Tests cover both the success path and at least one failure path.
- Diagnostics include deterministic locations when location data is available.
- The slice does not add runtime dependencies without a strong reason.
- `just test` passes.
- Any changed behavior is reflected in README or specs when user-facing.

## Risks

- TypeScript JSDoc association can be subtle. Keep scanner tests close to the
  supported declaration list and prefer the Compiler API over comment text
  heuristics.
- Markdown anchor generation may diverge from real GitHub anchors. v0.1 only
  needs the documented approximation, so tests should lock that behavior.
- Glob support is intentionally limited. Reject unsupported syntax early instead
  of partially emulating common glob libraries.
- CLI tests may need child process helpers. Keep them small and avoid depending
  on shell-specific behavior.
