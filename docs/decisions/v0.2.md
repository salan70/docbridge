# v0.2 Decisions

This document records the product and implementation-scope decisions for SpecLink
v0.2.

Detailed normative behavior lives in the feature spec:

- [LSP](../specs/lsp.md)
- [Diagnostics](../specs/diagnostics.md)

v0.1 behavior is unchanged. v0.2 is additive.

## Goal

v0.2 brings SpecLink into the editor. The same link graph that `speclink check`
validates becomes navigable and visible through a Language Server.

The deliverable is a Language Server, `speclink lsp`, that an editor connects to
and that answers four LSP features over the SpecLink link graph:

- `textDocument/publishDiagnostics`
- `textDocument/hover`
- `textDocument/definition`
- `textDocument/references`

The milestone is verified end to end in a real editor (VS Code) so the LSP
experience is real, not protocol-only.

## Scope

In scope:

- A Language Server process speaking LSP over stdio.
- The four features above.
- A minimal VS Code client extension, isolated in its own package, used to launch
  and verify the server.

Out of scope (deferred):

- Standalone single-binary distribution.
- npm publishing of the server.
- Editors other than VS Code as a verified target.
- Marketplace publishing of the VS Code extension.
- Incremental re-resolution and other performance work.
- Multi-root / monorepo workspaces.
- Code intelligence that other language servers already provide (types,
  completion, formatting). SpecLink only answers about the link graph.

## Why a Language Server, and why self-implemented

The README headline is "Bring Markdown into the LSP world." Hover, Definition,
and References are LSP methods by nature. A CLI-query surface would diverge from
the real usage (in-editor navigation) and force a rewrite later, so v0.2 builds
a real Language Server.

The transport (JSON-RPC over stdio, lifecycle, document sync) is implemented
without an LSP framework library. This keeps the server aligned with the
dependency policy in `CLAUDE.md`: minimal dependencies, Bun and the TypeScript
Compiler API first. The added surface is bounded and well specified; see
[LSP](../specs/lsp.md).

### Minimal-dependency policy exception

`CLAUDE.md` requires a dependency-light core. v0.2 introduces exactly one
controlled exception, and it does not touch the core or the server:

- The **VS Code client extension** depends on `vscode-languageclient` and the VS
  Code extension API. This dependency is required to run any LSP server inside VS
  Code; vanilla VS Code cannot launch an arbitrary LSP binary without a client
  extension.

This exception is isolated in a separate package (`editors/vscode/`). The server
(`speclink lsp`) and `src/core/` stay dependency-minimal and have no dependency
on `vscode-languageclient`. The core dependency policy is preserved; only the
editor-integration package, which is inherently editor-coupled, carries the
extra dependency.

### Partial pull-forward of editor integration

The README roadmap places editor integration at v0.4. v0.2 pulls forward a
**minimal** VS Code client extension, solely to verify the server in a real
editor. This is a deliberate, narrow pull-forward:

- v0.2 ships only the thin client glue needed to launch and exercise the server.
- Full editor integration (packaging, Marketplace, Cursor, Codex, MCP) remains
  v0.4.

## Architecture

### Project model

The server uses a whole-project model.

- On `initialize`, the server resolves the project root from `rootUri` /
  `workspaceFolders`. On `initialized`, it loads `speclink.config.json`, scans
  every file matched by the include globs from disk, and builds the full link
  graph.
- Open documents (including unsaved buffers) overlay their on-disk versions. The
  scanners already accept `(filePath, content)`, so the server feeds buffer text
  in place of disk content.
- On document change, the server re-resolves the whole project. A short debounce
  coalesces rapid edits.

A whole-project model is required because SpecLink diagnostics and References are
cross-file: backlink diagnostics and "find all code that implements this spec"
cannot be answered from a single open file.

Single-root only. Multi-root workspaces are out of scope for v0.2.

### Document sync

Full document sync (`TextDocumentSyncKind.Full`). The editor sends the full text
on open and on every change. This is the simplest correct model at this scale.

### Reusing the v0.1 engine

The v0.1 scanners and resolver are reused. The resolver currently emits only
diagnostics; v0.2 also exposes the resolved link graph (endpoint-to-endpoint
pairs with locations) as a queryable structure for navigation.

## Navigation behavior

### Trigger zones

Navigation fires on element text, not whole lines:

- Code: the declaration **name** identifier (for example, `login` in
  `export function login`).
- Markdown: the **heading text**.

This requires enriching the scanners to record ranges (name range, heading-text
range, and annotation target-string range) instead of single points. The same
ranges back the Definition target range and the References highlight range.

### Hover content

- Code to doc: render the linked Markdown section **inline** (heading plus body),
  cut from the target heading to the next heading at the same or a higher level.
  Deeper subsections are included. Fenced code blocks are not treated as
  headings.
- Doc to code: show the linked code endpoint plus its declaration signature line.
- One-to-many: concatenate the linked sections, separated.
- A loose length cap protects against very long sections.

### Definition and References (symmetric counterpart model)

SpecLink links are symmetric pairs, so Definition and References both traverse
the same counterpart relation:

- Definition: jump to the linked counterpart. Code to its doc section(s); doc to
  its code declaration(s). One-to-many returns multiple `Location`s; the editor
  shows a picker.
- References: list every counterpart linked to the element. This is most useful
  one-to-many (one spec, many implementing symbols).

### Navigation honors resolvable one-way links

Navigation follows any declared annotation whose target resolves to a real file
and anchor, regardless of whether the reverse backlink exists. Backlink
completeness is reported separately by diagnostics. This keeps the authoring loop
fluid: a freshly written `@doc` navigates immediately, while a diagnostic asks
for the missing `@code` backlink.

Targets that do not resolve (missing file or anchor) are never navigable; they
are diagnostics only.

## Diagnostics in the editor

The server publishes the existing v0.1 diagnostics through
`textDocument/publishDiagnostics`. The diagnostic computation is unchanged; v0.2
maps `SpecLinkDiagnostic` to the LSP `Diagnostic` shape:

- `severity`: error to `1`, warning to `2`.
- `range`: target-string range for link-target diagnostics; element range for
  declaration/heading diagnostics; the whole line as a fallback when no range is
  available.
- `code`: the SpecLink diagnostic code.

Diagnostic codes are defined normatively in
[Diagnostics](../specs/diagnostics.md). v0.2 adds no new codes.

Push model: the server publishes for open documents. The full graph is held in
memory, so open documents receive correct cross-file diagnostics.

## Positions and paths

- Position encoding is LSP-default UTF-16 code units. SpecLink line and column
  data is derived from JavaScript string indexing, which is already UTF-16, so
  this aligns without conversion.
- LSP `line` / `character` are 0-based; SpecLink `line` / `column` are 1-based.
  The server converts at the boundary.
- `file://` URIs convert to project-root-relative paths. Windows path specifics
  are out of scope for v0.2.

## CLI

`speclink check` is unchanged. v0.2 adds:

```sh
speclink lsp
```

`lsp` runs the Language Server over stdio. It takes no options; the project root
comes from the LSP `initialize` request.

## Verification

- Automated: a protocol conformance test drives the server as a child process
  (`initialize` -> `didOpen` -> request/response) and asserts LSP responses.
  This is the CI foundation.
- Manual: the VS Code client extension runs under the Extension Development Host
  (F5), connected to `speclink lsp`, to confirm Diagnostics, Hover, Definition,
  and References against a TypeScript and Markdown file.

## Future Scope

Future versions may add:

- standalone single-binary distribution (`bun build --compile`)
- npm publishing of the server
- additional verified editors (Neovim and others)
- VS Code Marketplace publishing
- incremental re-resolution and large-project performance work
- multi-root / monorepo workspaces
- richer doc-to-code hover (full signature, doc comment)
- the v0.3 and v0.4 roadmap items (context generation, MCP server, Cursor and
  Codex integration)
