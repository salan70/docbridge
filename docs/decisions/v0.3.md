# v0.3 Decisions

This document records the product and implementation-scope decisions for
SpecLink v0.3.

Detailed normative behavior lives in the feature spec:

- [CLI — Context Command](../specs/cli.md#context-command)

v0.1 and v0.2 behavior is unchanged. v0.3 is additive.

## Goal

v0.3 makes the SpecLink link graph consumable by AI agents. The link graph
that `speclink check` validates and `speclink related` enumerates becomes
injectable *content*: the specification sections and code declarations an
agent should have read before editing a linked file.

v0.3 establishes the content and its format. v0.4's MCP server is a delivery
channel layered on top of the same output; it is not part of v0.3.

## Use cases

In priority order:

1. **Edit-time context injection.** An agent fetches the linked counterpart
   content of the files it edits, so it reconciles the change against the
   relevant specification without extra file discovery.
2. **Gate and review enhancement.** When `related --gate` flags counterparts
   that were not updated, the triage step fetches their content to judge
   whether the specification and the code have actually diverged.

Both consume the same operation — "file set in, counterpart content out" — so
v0.3 ships it as one command.

A note on hook timing (learned while dogfooding): Claude Code delivers a
`PreToolUse` hook's `additionalContext` next to the tool result, i.e. *after*
the edit runs, so a hook cannot enforce read-*before*-editing. The dogfooded
hook is therefore a `PostToolUse` hook that surfaces the counterpart content
immediately after the edit, while the agent can still reconcile it before
moving on. A skill or agent that calls `speclink context` directly can still
read the counterpart strictly before editing; the limitation is specific to
automatic hook delivery.

## The `context` command

```sh
speclink context [--root <path>] [--json] [--stdin] [files...]
```

Decisions, with the reasoning:

- **A new command, not a `related` flag.** `related` is a lightweight path
  enumerator with a gate exit code, consumed by hooks and CI; mixing body
  content into its output would muddy that contract. The two compose instead:
  the gate flags counterparts, `context` fetches what they say.
- **File-level input only.** Both use cases feed file path sets (the files an
  agent edits, `git diff --name-only` output). No `file#fragment` narrowing:
  there is no caller for it in v0.3, and adding it later is backward
  compatible.
- **One hop only.** Direct counterparts carry the "must read before editing"
  semantics, match the gate's reporting, and keep output size predictable.
  Transitive expansion (`--depth`) can be added compatibly if a real need
  appears.
- **Full content.** Doc counterparts contribute the full Markdown section
  (promoted from the LSP hover section extractor into `src/core/`); code
  counterparts contribute the full declaration including JSDoc, backed by a
  new `declarationRange` recorded by the TypeScript scanner. Reference-only
  output would make the command redundant with `related`, and signature-only
  output hides spec/implementation divergence from doc editors.
- **Markdown by default, `--json` for tooling.** The primary consumption is
  direct injection into an agent prompt, so the default output is plain
  Markdown. `--json` (with `schemas/context-output.schema.json`) is the
  foundation the v0.4 MCP server will wrap.
- **Best-effort, exit 0.** The command runs around edits, on trees that are
  often temporarily broken. A strict mode would withhold all context because one
  link is broken; silent skipping would hide typos. So: emit every block that
  resolves, report check diagnostics located in the input files (stderr, or
  the `diagnostics` field), and reserve non-zero exits for invocation and
  configuration errors. Correctness enforcement stays with `check` and
  `related --gate`.

## Scope

In scope:

- The `speclink context` command as specified above.
- AI integration: dogfooding `context` in this repository's agent hooks
  (Claude Code and Codex), integration recipes in `docs/`, and reusable hook
  scripts in `examples/`.
- Two distributable skills under `templates/skills/`, also installed in this
  repository: `speclink-annotate` (creating links) and `speclink-sync`
  (triaging gate findings with `context`). Codex equivalents follow once the
  Claude versions settle.

Out of scope (deferred or rejected):

- **MCP server** — v0.4; `context --json` is its foundation.
- **LSP exposure of context** (custom request) — considered and rejected:
  v0.3's consumers are hooks and skills, which call the CLI. Editor delivery
  channels are a v0.4 design alongside MCP.
- **`file#fragment`-level input** and **`--depth` / transitive resolution** —
  no v0.3 caller; both are backward-compatible extensions.
- **Token-budget controls** (`--max-tokens`) — deferred until output size is a
  measured problem; declaration- and section-bounded content keeps it small.
- **Incremental re-resolution and performance work** — unchanged from v0.2;
  `context` performs one whole-project scan per invocation.
- **Plugin or marketplace distribution of skills** — copy-based installation
  only; packaging is premature before the skill content stabilizes.
- **npm publishing, single-binary distribution, editors other than VS Code** —
  v0.2 deferrals stay deferred.

## Future Scope

v0.4 builds the delivery channels on top of `context`: an MCP server exposing
the same data as tools, and editor/agent integrations (Claude Code, Cursor,
Zed, Codex). The dogfooding feedback from v0.3's hooks and skills — which hook
events benefit, how much context is useful — is the input to that design.
