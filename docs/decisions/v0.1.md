# v0.1 Decisions

This document records the product and implementation-scope decisions for SpecLink v0.1.

Detailed normative behavior lives in the feature specs:

- [Configuration](../specs/configuration.md)
- [Scanning](../specs/scanning.md)
- [Annotations](../specs/annotations.md)
- [Link Resolution](../specs/link-resolution.md)
- [Diagnostics](../specs/diagnostics.md)
- [CLI](../specs/cli.md)

## Goal

v0.1 should be usable on this repository, not only on a fixture.

The first deliverable remains:

```sh
speclink check --root examples/typescript
```

The implementation should also support small TypeScript and Markdown projects that follow the v0.1 configuration and annotation rules.

## Documentation Model

`docs/specs/` contains current, code-linkable specifications.

`docs/decisions/` contains decision records and rationale. Decision records are not part of the root SpecLink docs include.

The root repository config includes:

```json
{
  "include": {
    "code": ["src/**/*.ts"],
    "docs": ["docs/specs/**/*.md"]
  }
}
```

`examples/typescript` is treated as a separate project root and has its own config.

## Configuration

v0.1 includes a strict JSON config file named `speclink.config.json`.

When no config file exists, SpecLink uses:

```json
{
  "include": {
    "code": ["src/**/*.ts"],
    "docs": ["docs/**/*.md"]
  }
}
```

When a config file exists, `include.code` and `include.docs` are required. Unknown keys and invalid values are diagnostics.

`$schema` is allowed as an optional string for editor tooling. SpecLink does not fetch or validate the schema URL in v0.1.

## Link Model

v0.1 uses explicit bidirectional annotations:

- TypeScript `@doc` creates a `code -> doc` link.
- Markdown `@code` creates a `doc -> code` link.
- A valid relationship requires the same pair in both directions.

One-to-many and many-to-many relationships are supported as multiple independent link pairs.

The public target format is:

```text
file#fragment
```

File paths are project-root-relative and case-sensitive.

## TypeScript Scope

`@doc` annotations are supported only in JSDoc comments that the TypeScript Compiler API associates with declarations.

Supported declarations are top-level exported `function`, `class`, `abstract class`, `interface`, `type`, `const` with one declarator, `enum`, `const enum`, named default `function`, named default `class`, and `declare` forms of supported declarations inside `.ts` files.

Class members, nested symbols, namespace/module declarations, re-exports, anonymous default exports, and `.d.ts` files are out of scope for v0.1.

## Markdown Scope

`@code` annotations are standalone HTML comments attached to the next ATX heading.

Markdown anchors use a Unicode-aware GitHub-style slug approximation. Duplicate anchors in the same file are errors.

Setext headings are out of scope for v0.1. Fenced code blocks are recognized so examples do not produce links.

## Diagnostics

v0.1 uses errors for invalid config, invalid targets, missing managed files, missing doc anchors, missing backlinks, duplicate doc anchors, duplicate code endpoints, TypeScript parse errors, and file read errors.

Warnings are used for duplicate annotations, dangling Markdown annotations, unsupported annotated declarations, and audit-only undocumented endpoints.

`orphan_doc`, `one_way_link`, and standalone code symbol existence diagnostics are out of scope for v0.1.

## Future Scope

Future versions may add:

- explicit `exclude` config
- an escape hatch for built-in ignores
- `.gitignore` support
- `.tsx` and `.d.ts` support
- Setext heading support
- file-level links
- URL-decoded fragments
- richer code endpoint fragments for overloaded or merged declarations
- `orphan_doc`
- LSP features such as hover, definition, and references
