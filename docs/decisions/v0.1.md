# v0.1 Decisions

This document records the initial scope decisions for SpecLink v0.1.

## Primary Slice

The first deliverable is:

```sh
speclink check --root examples/basic
```

The command should scan TypeScript and Markdown files, resolve declared links, and report diagnostics.

Development uses `flake.nix` and `direnv`. Contributors should enable the development environment with:

```sh
direnv allow
```

They can also enter the development environment manually with:

```sh
nix develop
```

Project tasks are run with `just`:

```sh
just --list
just check-example
just test
just build
```

## TypeScript Scope

`@doc` annotations are supported only in JSDoc comments.

Supported symbols are top-level exports:

- `function`
- `class`
- `interface`
- `type`
- `const`

The TypeScript fragment is the declaration name:

```text
src/auth/login.ts#login
```

Class members and nested symbols are out of scope for v0.1.

## Markdown Scope

`@code` annotations are written in HTML comments and attached to the next heading:

```md
<!-- @code src/auth/login.ts#login -->
## Login Spec
```

Markdown anchors are generated from heading text using a GitHub-style slug:

```text
## Login Spec -> login-spec
```

Duplicate Markdown anchors are errors only within the same file. The same anchor in different files is allowed.

## Link Resolution

The public link format is:

```text
file#fragment
```

Internally, targets should be modeled as structured data:

```ts
type LinkTarget = {
  filePath: string;
  fragment: string;
};
```

A valid bidirectional link requires both sides to agree:

- TypeScript `@doc` points to a Markdown section
- Markdown `@code` points back to the TypeScript symbol

`one_way_link` is emitted only when the referenced target exists. Missing file or missing fragment diagnostics take precedence.

## Scan Defaults

Without configuration, SpecLink scans:

- `src/**/*.ts`
- `docs/**/*.md`

The CLI supports `--root` to scan a different project root:

```sh
speclink check --root examples/basic
```

No configuration file is planned for v0.1.

## Diagnostics

Default `speclink check` diagnostics:

- `doc_file_not_found`
- `doc_anchor_not_found`
- `code_file_not_found`
- `code_symbol_not_found`
- `duplicate_doc_anchor`
- `one_way_link`

Audit diagnostics are available only with:

```sh
speclink check --audit
```

Audit diagnostics:

- `undocumented_symbol`
- `orphan_doc`

## Output

The default output is human-readable text.

`--json` emits an object:

```json
{
  "diagnostics": [],
  "summary": {
    "errors": 0,
    "warnings": 0
  }
}
```

## Exit Codes

- Exit `1` when any error exists.
- Exit `0` when diagnostics contain only warnings or no diagnostics.
